# ABACUS Post-PR #522 Delta Candidates

Scope: smallest executable ABACUS-side deltas after PR #522, constrained to repository evidence already present in CODEX plus the previously identified ABACUS evidence: federation telemetry dashboard, `build_federation_dashboard.py`, Pages deployment guidance, and existing telemetry concepts.

Non-goals:

- No new federation architecture.
- No federation runtime.
- No bootstrap repositories.
- No patch-bundle generation.

## Evidence constraints

- CODEX already treats ABACUS as the governance, metrics-source, and validation participant while CODEX owns federation rollup and cross-repository metrics.
- CODEX maps ABACUS renderer and deployment lanes to existing dashboard and Pages workflows, not to a new runtime.
- CODEX already has a static federation bridge dashboard and progress tracker that distinguish staged templates from live automation.
- CODEX already records telemetry concepts using source/wave/timestamp/metric/value events and summarized metrics.
- CODEX Pages deployment stages existing `docs/` outputs into `_site`.

## Delta Candidate A

### Description

Add a static federation telemetry section/card to the existing ABACUS federation telemetry dashboard generated by `build_federation_dashboard.py`. The section should render ABACUS-owned governance/metrics-source/validation status and link to the existing Pages-published dashboard location. The data should come from existing ABACUS telemetry concepts only; if a live feed is unavailable, render the current known telemetry summary as a deterministic static snapshot.

### Files touched

- `build_federation_dashboard.py`
- Existing generated federation telemetry dashboard HTML artifact
- Existing Pages documentation/link index only if the dashboard is not already linked

### Risk

Low. This reuses the existing dashboard builder, existing dashboard artifact, existing telemetry vocabulary, and existing Pages surface. It avoids runtime creation, workflow expansion, repository bootstrap, and generated patch bundles.

### Expected value

High. It creates the first mergeable, visible ABACUS-side federation delta after PR #522 while preserving the current CODEX/ABACUS separation of concerns.

## Delta Candidate B

### Description

Add a deterministic telemetry input fixture or summary JSON consumed by `build_federation_dashboard.py`, using the existing ABACUS telemetry schema/concepts. The dashboard build should read the fixture and render the same static federation telemetry table/card as Candidate A. This is still a dashboard-builder delta, not a runtime ingestion feature.

### Files touched

- `build_federation_dashboard.py`
- One small existing telemetry data file if available, or one new minimal telemetry summary file if no existing file can be reused
- Existing generated federation telemetry dashboard HTML artifact

### Risk

Medium-low. The executable path is still the existing dashboard builder, but the candidate may require one small data artifact if ABACUS has no reusable telemetry summary file. Keep schema validation local to the builder and avoid adding CI jobs.

### Expected value

Medium-high. It makes the dashboard output less hand-coded and easier to validate after PR #522, while still avoiding new federation runtime, new workflows, or new repositories.

## Delta Candidate C

### Description

Add one minimal Pages check or documentation line proving that the ABACUS federation telemetry dashboard is included in the existing Pages deployment. Do not add a new workflow; use the existing Pages deployment guidance and only extend an existing check/list if the dashboard is currently unreferenced.

### Files touched

- Existing Pages documentation/index/check file that already governs published dashboard paths
- Existing generated federation telemetry dashboard HTML artifact only if the link target needs a title or anchor update

### Risk

Low technically, but lower functional value than A or B. This validates discoverability and deployment wiring without increasing executable dashboard behavior.

### Expected value

Medium. It reduces post-merge ambiguity by showing where the dashboard appears on Pages, but it does not materially improve telemetry rendering unless paired with A or B.

## Ranking

1. **Recommended: Delta Candidate A** — smallest executable value: one existing builder, one existing dashboard, existing telemetry concepts, and existing Pages surface.
2. **Optional: Delta Candidate B** — useful if ABACUS needs a stable data handoff into the builder, but only after confirming an existing telemetry file cannot be reused.
3. **Future: Delta Candidate C** — keep as a follow-up or companion if Pages discoverability is missing; do not prioritize over executable dashboard content.
