# Contributing

## Working Agreement

Contributions should be scoped, traceable, and reviewable. Prefer small pull requests that identify the affected governance domain, validation evidence, and follow-up work.

## Branches and Commits

- Use feature branches for new work.
- Do not force-push shared branches unless maintainers explicitly approve.
- Use clear commit messages that describe the intent and affected artifact.

## Pull Requests

Pull requests should use the governance bootstrap template and include:

- Summary of changes.
- Scope and ownership.
- RTM, DMAIC, ADR, or governance linkage when relevant.
- Exact validation commands and results.

## Documentation Standards

- Keep governance documents concise and auditable.
- Prefer tables for ownership, traceability, and decision records.
- Preserve source-of-truth references when moving or summarizing content.

## Review Standards

Reviewers should check scope control, evidence quality, security considerations, and whether the change introduces new governance obligations.
